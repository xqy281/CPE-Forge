# 📄 产品需求文档 (PRD)：CPE研发效能与人才画像AIGC分析平台

## 1. 产品概述
本平台旨在通过AIGC技术，深度解析CPE嵌入式软件团队过去一年以上的Excel周报数据。通过清洗去噪、长文本大模型推理，将枯燥的周报转化为直观的“双层能力雷达图”和“成长时间轴”，并提供基于全量周报上下文的连续对话FAQ功能。系统采用**全员公开透明**的权限设计，打造极客化、游戏化的团队数字孪生看板。

## 2. 系统架构与数据管线 (Data Pipeline)

### 2.1 智能识别与过滤 (Auto-Discovery)
*   **需求描述**：系统需支持批量上传/拉取包含周报的Excel附件。
*   **处理逻辑**：不依赖文件名，后端读取Excel表头，若包含预设特征列（如“任务描述”、“进度”、“难点”、“下周计划”等）命中率超过设定阈值，则判定为有效周报，剔除无关财务/行政表格。

### 🔄 PRD 修订版：2.2 数据清洗与去重 (Noise Reduction - Content-Based)
*   **需求描述**：消除因“包含前两周记录”制度带来的数据冗余，同时**必须避免**因员工忘记修改Sheet名或文件名而导致的历史周报误删。去重判定需深入到周报的实际文本内容。
*   **处理逻辑**：
    1.  **数据展平与文本拼接 (Data Flattening)**：
        *   以“员工姓名”为聚合主键，提取该员工名下所有Excel文件中的所有Sheet。
        *   将每个Sheet内的核心业务字段（如：任务描述、进度、难点分析、下周计划）按行拼接成一个完整的“周报纯文本字符串”。
    2.  **内容相似度计算 (Similarity Calculation)**：
        *   在后端引入轻量级的文本相似度算法（建议使用 `TF-IDF + 余弦相似度` 或 `SimHash` 算法，计算速度快，适合长文本比对）。
        *   对该员工名下的所有“周报纯文本字符串”进行两两比对，计算相似度百分比。
    3.  **94% 阈值去重规则 (94% Threshold Deduplication)**：
        *   **判定重复**：当两个或多个Sheet的文本相似度 **≥ 94%** 时，系统判定它们为“重复提交的冗余周报”（例如：仅修改了几个错别字、标点符号，或完全未修改）。
        *   **保留策略 (Survivor Rule)**：在被判定为重复的Sheet组中，系统需执行“优胜劣汰”：
            *   优先保留 **文件最后修改时间 (Last Modified Time) 最新** 的那份数据。
            *   若时间戳不可靠，则保留 **有效字符数最多** 的那份（通常意味着员工在最后时刻补充了少量细节）。
            *   其余 ≥ 94% 相似度的Sheet直接从处理管线中丢弃。
    4.  **时间线容错重构 (Timeline Reconstruction)**：
        *   由于Sheet名已经不可靠（可能全是“Sheet1”或“第12周”），对于通过了去重校验（相似度 < 94%）的有效周报，后端脚本需优先通过正则匹配提取表头或内容中的“实际汇报日期/周期”。
        *   若内容中无日期，则降级使用该Excel文件的“创建/修改时间”作为时间轴锚点，按时间正序重新排列，最终生成用于喂给大模型的 Markdown 长文本。

### 2.3 Token极致压缩与预检 (Token Optimization & Pre-flight)
*   **需求描述**：将Excel转化为大模型友好的格式，并在请求前提供Token消耗预估。
*   **处理逻辑**：
    1.  **格式转换**：将清洗后的个人年度周报按时间正序，转换为极简的 **Markdown 表格** 格式（`| 日期 | 任务 | 进度 | 难点 |`）。
    2.  **Token计算**：后端调用分词器（如 `tiktoken`）计算生成的Markdown字符串的精确Token数。
    3.  **UI预检看板**：在Web端发起AIGC分析前，展示预估Token量。设置水位线（如：绿色<50k，黄色50k-100k，红色>模型上限），避免无效请求崩溃。

## 3. 核心功能模块需求 (Core Features)

### 3.1 模块一：双层能力画像 (Dual-Layer Profile Dashboard)
*   **外层雷达图（技术广度与投入度）**：
    *   **维度**：系统平台、底层驱动、上层软件应用、无线通信、SQA。
    *   **渲染逻辑**：AI根据周报任务归属统计各维度权重，前端渲染为多角面积雷达图。
*   **内层内核（工程素养成熟度）**：
    *   **维度**：求真 (Truth-seeking) -> 务实 (Pragmatic) -> 严谨 (Rigorous)。
    *   **AI提取规则**：
        *   *求真*：识别“定位根因、阅读手册、排查底层逻辑”等溯源行为。
        *   *务实*：识别“任务闭环、按时交付、临时规避/替代方案”等交付行为。
        *   *严谨*：识别“边界测试、内存泄漏排查、代码重构、自动化脚本”等质量行为。
    *   **渲染逻辑**：在雷达图中心以颜色深度或等级（Lv1-Lv5）展示基石状态。

### 3.2 模块二：成长变化与时间轴分析 (Growth & Timeline Analysis)
*   **功能1：问题闭环追踪 (Closed-loop Tracking)** *【核心高亮】*
    *   **AI提取规则**：在时间线上提取“未决问题/难点”，并在后续周报中持续追踪该实体的下文，直到提取出“最终解决方案”。
    *   **UI呈现**：在时间轴上形成“故事线”卡片，点击可查看该问题跨越N周的解决全过程。
*   **功能2：剥洋葱式深度递进识别 (Recursive Logic Analysis)**
    *   **AI提取规则**：分析跨周同类任务的排查路径。
        *   *正向（剥洋葱）*：排查范围逐层缩小，有明确因果推导（如：现象->日志->模块->代码行），打上“深度递进/求真严谨”标签。
        *   *负向（乱试验）*：变量发散，盲目替换（如：换电源->刷固件->换主板），打上“横向试错/缺乏务实”提示标签。

### 3.3 模块三：全量上下文 FAQ 智能对话 (Full-Context AI Chat)
*   **技术路线**：**弃用RAG**。直接将该员工（或指定团队范围）清洗后的全量Markdown周报作为System Prompt的背景上下文（Context）注入。
*   **交互设计**：
    *   提供类似 ChatGPT 的 Web Chat 界面。
    *   **连续对话支持**：前端/后端需维护对话历史（Message History Array），每次请求携带之前的多轮问答上下文。
    *   **应用场景**：主管追问某专项进度细节、新员工查询历史踩坑记录（如“去年张三是怎么解决高通PCIe枚举失败的？”）。

### 3.4 模块四：Web UI 视图与权限 (Web UI & Permissions)
*   **权限设定**：**完全公开**。所有人可查看团队全员数据。
*   **页面规划**：
    1.  **数据调度台 (Data Console)**：上传Excel、执行清洗、查看Token预检仪表盘、触发AIGC生成。
    2.  **团队大盘 (Team Dashboard)**：CPE团队整体技术投入热力图、人员分布。
    3.  **英雄榜/花名册 (Roster)**：全员“球星卡”列表，展示核心雷达缩略图。
    4.  **个人详情页 (Player Detail)**：双层雷达图放大版、成长时间轴（问题闭环故事线）、个人专属FAQ对话框。

## 4. 提示词工程指导 (Prompt Engineering Guidelines)
*(交由大模型算法工程师参考)*
*   **System Prompt 结构**：
    1.  **角色设定**：资深CPE嵌入式软件研发总监。
    2.  **任务目标**：对提供的Markdown格式周报进行量化提取与逻辑分析。
    3.  **输出格式**：强制要求输出标准化的 JSON 格式，以便前端直接解析渲染（包含 `radar_outer`, `radar_inner`, `closed_loop_issues`, `growth_analysis` 等字段）。
    4.  **判定Few-Shot示例**：在Prompt中提供1-2个“剥洋葱”和“乱试验”的文本对比示例，校准大模型的判定尺度。

## 5. 非功能性需求与技术建议
*   **大模型选型**：由于采用全量上下文注入（非RAG），必须选择支持长上下文（至少128K，推荐200K+）且逻辑推理能力极强的模型（如 Claude 3.5 Sonnet, GPT-4o, 或国产的 DeepSeek-V2/V3 等）。
*   **响应时间管理**：长文本推理耗时较长（可能达数十秒至数分钟），Web端必须设计优雅的 Loading 动画（如“正在分析第32周底层驱动数据...”）或采用异步任务队列（Celery/RabbitMQ）+ WebSocket 推送结果。
